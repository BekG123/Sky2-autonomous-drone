DShot600 PIO Driver

A hardware-level DShot600 motor control driver for the Raspberry Pi Pico W. Uses the RP2040's programmable I/O (PIO) state machines to generate four perfectly phase-locked DShot waveforms simultaneously — with bit-accurate timing that is completely independent of CPU load or Python execution.

4.8 MHz PIO clock
1.667 µs Per bit
4 Phase-locked motors
1 kHz Update rate
120 MHz CPU clock

What this is

DShot is a digital serial protocol used to control brushless motor ESCs (Electronic Speed Controllers) on drones and other robotics platforms. Unlike analog PWM, it sends a 16-bit packet per frame — 11 bits of throttle, 1 bit telemetry request, and 4 bits of CRC — with no calibration required and corruption detection built in.

This driver runs the entire waveform generation in the RP2040's PIO hardware. The CPU's only job is to write a 32-bit value into a hardware FIFO register once per millisecond. Everything else — bit timing, signal toggling, inter-packet gaps — happens autonomously at the PIO clock rate, with zero CPU involvement between frames.

Key properties

  • Hardware-level bit timing. Each bit takes exactly 8 PIO cycles (1.667 µs at 4.8 MHz). Timing does not depend on Python execution speed.
  • Phase-locked four-motor operation. All four state machines start on the same CPU clock cycle and remain in lockstep indefinitely — data-independent cycle counts guarantee this.
  • Gapless stream via 1 kHz hardware timer. A periodic timer ISR feeds the FIFOs continuously, so print(), garbage collection, or any Python latency cannot create signal gaps that would disarm the ESCs.
  • Direct FIFO writes. mem32 writes bypass MicroPython's state machine API, removing per-call overhead and ensuring the four writes land within a handful of CPU cycles of each other.

Quick Start

Flash the Pico W, connect your ESCs to GPIO 6, 5, 4, and 3, then run:

  1. 1

    Copy the driver to your Pico

    Transfer dshot_parallel.py to the Pico W via Thonny or mpremote. No external libraries are required — only the MicroPython standard library and the rp2 module built into RP2040 firmware.

  2. 2

    Wire your ESCs

    Connect each ESC's signal wire to the corresponding GPIO pin. No level shifter needed — ESCs accept 3.3 V DShot logic from the Pico directly.

    State MachineGPIOMotor
    SM 0GPIO 6Motor 1
    SM 1GPIO 5Motor 2
    SM 2GPIO 4Motor 3
    SM 3GPIO 3Motor 4
  3. 3

    Run with props removed

    Always bench-test with propellers removed. The script arms the ESCs at throttle 0 for 2 seconds, then ramps from 48 (0 % effective throttle) to your configured maximum.

    python 
    import rp2, machine
from machine import Pin, mem32
import time

machine.freq(120_000_000)
MOTOR_PINS   = [6, 5, 4, 3]
DSHOT_FREQ   = 4_800_000   # DShot600
PIO0_CTRL    = 0x50200000
SM_TXFIFO    = [0x50200010, 0x50200014, 0x50200018, 0x5020001c]

# After defining the dshot() PIO program and make_packet() ...

dshot_throttle = 0

def _dshot_ticker(timer):
    packet = make_packet(dshot_throttle) << 16
    for addr in SM_TXFIFO:
        mem32[addr] = packet

def main():
    global dshot_throttle
    state_machines = [
        rp2.StateMachine(i, dshot, freq=DSHOT_FREQ, sideset_base=Pin(MOTOR_PINS[i]))
        for i in range(4)
    ]
    for sm in state_machines:
        sm.restart()
    mem32[PIO0_CTRL] = 0b1111          # start all 4 SMs on same cycle

    timer = machine.Timer()
    timer.init(freq=1000, mode=machine.Timer.PERIODIC, callback=_dshot_ticker)

    dshot_throttle = 0
    time.sleep(2)                       # arm phase — timer streams 0 continuously

    for t in range(48, 600):
        dshot_throttle = t
        time.sleep_ms(50)

System Architecture

The driver is organized into three independent layers. The key design insight is that the physical signal — what the ESC actually sees — is generated entirely in PIO hardware, not Python. Python only decides what value to send; the PIO decides how to send it.

LayerRuns onRateJob
Main loop RP2040 CPU core 0 Application-defined Updates the dshot_throttle global variable. Can sleep, print, or compute without affecting signal quality.
1 kHz timer ISR RP2040 CPU (IRQ) 1,000 Hz Calls _dshot_ticker() once per millisecond, writing the current throttle packet to all four TX FIFOs via mem32.
PIO state machines PIO0 hardware (SM 0–3) 4.8 MHz Each SM runs the dshot() PIO program autonomously. Pulls packets from its FIFO and shifts them out one bit every 8 cycles (~1.667 µs). Drives the GPIO directly via .side() instructions.
ℹ

The PIO does not use the CPU between frames. Once a packet is in the FIFO, the state machine runs entirely on its own clock divider (derived from the 120 MHz system clock). A Python GC pause, a blocked print(), or any other CPU latency cannot affect the signal that is already in flight.

Because all four SMs run the same program at the same clock frequency and are started simultaneously with a single mem32[PIO0_CTRL] = 0b1111 write, they remain phase-locked for the lifetime of the program. Both the ONE path (bit = 1) and the ZERO path (bit = 0) through the PIO program take exactly 8 cycles, so there is no drift regardless of the data being sent.

DShot Protocol

DShot is a unidirectional digital protocol (the extended DShot-EDT variant adds bidirectional telemetry, but that is not implemented here). Each frame is a 16-bit packet clocked out MSB-first with no gaps between bits.

Packet structure

FieldBitsPositionDescription
Throttle 11 bits 15 → 5 Motor command. 0 = disarm signal. 1–47 = reserved special commands (do not send as throttle). 48–2047 = throttle range (0 % → 100 %).
Telemetry 1 bit 4 Set to 1 to request ESC telemetry on the next frame. Normally 0.
CRC 4 bits 3 → 0 Cyclic Redundancy Check. Computed as the XOR of the three upper nibbles of the 12-bit (throttle + telemetry) value, masked to 4 bits. Allows the ESC to detect corrupted frames.

Throttle range

Value rangeMeaning
0Disarm / idle signal. Sent continuously during arming.
1 – 47Reserved special commands. Interpreted by the ESC as digital commands (beep, motor direction, 3D mode, etc.). Never send these as throttle — they will trigger ESC actions, not spin the motor.
48 – 2047Throttle. 48 = 0 % thrust, 2047 = 100 % thrust. Always start ramps from 48.
⚠

Never start a throttle ramp from 1. Values 1–47 are special ESC commands. Sending them sequentially during a ramp triggers beeps, mode changes, or other ESC behavior — which is the most common cause of erratic motor behavior during first bring-up. Always start at 48.

Bit Timing

DShot encodes each bit as a fixed-width pulse. The width of the HIGH portion determines whether the bit is a 1 or 0. The total bit period is always the same — only the duty cycle changes.

ProtocolBit periodT1H (bit = 1)T0H (bit = 0)PIO frequencyCycles / bit
DShot 150 6.667 µs 5.00 µs (75 %) 2.50 µs (37.5 %) 1.2 MHz 8
DShot 300 3.333 µs 2.50 µs (75 %) 1.25 µs (37.5 %) 2.4 MHz 8
DShot 600 1.667 µs 1.25 µs (75 %) 0.625 µs (37.5 %) 4.8 MHz 8

The PIO program uses 8 cycles per bit for all three speeds. Switching between DShot variants only requires changing DSHOT_FREQUENCY — no changes to the PIO assembly. All three frequencies divide evenly from 120 MHz (dividers of 100, 50, and 25 respectively), so there is no fractional clock jitter on any speed.

DShot600 bit encoding — bit 1 stays HIGH for 75 % of the period; bit 0 for 37.5 %

Packet Construction

make_packet(throttle, telemetry=False) builds the 16-bit DShot frame in three steps.

python 
def make_packet(throttle, telemetry=False):
    # Step 1: shift throttle left to make room for the telemetry bit
    packet = (throttle << 1) | (1 if telemetry else 0)

    # Step 2: compute CRC — XOR the three upper nibbles, keep lower 4 bits
    crc = (packet ^ (packet >> 4) ^ (packet >> 8)) & 0x0F

    # Step 3: append CRC in the lower 4 bits
    return (packet << 4) | crc

Worked example: throttle = 1024

StepOperationResult (hex)Result (binary)
Throttle 1024 0x400 100 0000 0000
Shift + telemetry 1024 << 1 | 0 0x800 1000 0000 0000
CRC nibbles 0x800 ^ 0x080 ^ 0x008 0x888 1000 1000 1000
Mask to 4 bits 0x888 & 0x0F 0x8 1000
Final packet (0x800 << 4) | 0x8 0x8008 1000 0000 0000 1000

The 16-bit packet 0x8008 is then left-shifted 16 positions before being written to the TX FIFO, placing the packet in the upper half of the 32-bit OSR. When the PIO shifts out bits using SHIFT_LEFT, it reads from MSB first — so bit 15 of the packet (the throttle MSB) exits the GPIO first, as DShot requires.

python 
def send_dshot_all(throttle):
    packet = make_packet(throttle) << 16   # 0x8008 → 0x80080000
    for addr in SM_TXFIFO:
        mem32[addr] = packet               # write directly to each TXFIFO register

PIO Program

The dshot() function is compiled by the @rp2.asm_pio decorator into a native PIO state machine program. It runs entirely on the PIO hardware, independent of the CPU.

python (PIO assembly) 
@rp2.asm_pio(
    out_shiftdir=rp2.PIO.SHIFT_LEFT,   # MSB-first — DShot requires this
    autopull=False,
    set_init=rp2.PIO.OUT_LOW,
    out_init=rp2.PIO.OUT_LOW,
    sideset_init=rp2.PIO.OUT_LOW,
)
def dshot():
    pull(block)             # wait for a packet in the FIFO; signal stays LOW
    set(x, 15)              # load bit counter: 16 bits (counts 15 down to 0)

    label("BITLOOP")
    out(y, 1).side(1) [1]          # shift 1 bit into y, pull signal HIGH — 2 cycles
    jmp(not_y, "ZERO").side(1)     # if bit=0, jump to ZERO; stays HIGH — 1 cycle

    # --- bit = 1 path (6 cycles HIGH, 2 cycles LOW) ---
    nop().side(1) [2]              # hold HIGH — 3 cycles
    jmp(x_dec, "BITLOOP").side(0) [1]  # loop if bits remain, pull LOW — 2 cycles
    jmp("DONE")                    # only reached on last bit if it was 1

    # --- bit = 0 path (3 cycles HIGH, 5 cycles LOW) ---
    label("ZERO")
    nop().side(0) [2]              # hold LOW — 3 cycles
    jmp(x_dec, "BITLOOP").side(0) [1]  # loop if bits remain, stay LOW — 2 cycles

    label("DONE")
    nop().side(0) [1]              # inter-packet LOW gap — 2 cycles, then wrap to pull()

Cycle Breakdown

The critical property of this PIO program is that both the bit-1 path and the bit-0 path through the loop take exactly 8 cycles. This data-independence is what keeps four state machines in perfect lockstep — there is no timing condition that could cause one SM to run faster or slower than another based on the data being sent.

InstructionCyclesSignalPath
out(y, 1).side(1) [1]2HIGHBoth
jmp(not_y, "ZERO").side(1)1HIGHBoth (jump taken if bit = 0)
nop().side(1) [2]3HIGHBit = 1 only
jmp(x_dec, "BITLOOP").side(0) [1]2LOWBit = 1 only
nop().side(0) [2]3LOWBit = 0 only
jmp(x_dec, "BITLOOP").side(0) [1]2LOWBit = 0 only
HIGH cyclesLOW cyclesTotalDuty cycle
Bit = 1 2 + 1 + 3 = 6 2 8 6/8 = 75 % ✓
Bit = 0 2 + 1 = 3 3 + 2 = 5 8 3/8 = 37.5 % ✓
ℹ

Why jmp("DONE") is necessary. When the last bit (x = 0) is a 1, jmp(x_dec, "BITLOOP") does not jump (x was already 0 before decrement). Without jmp("DONE"), execution falls through into the ZERO label body, runs two more instructions, and then sees x = 0xFFFFFFFF (non-zero after wrap) — causing the bit loop to restart with a corrupted counter and spurious signal. The jmp("DONE") after the bit-1 loop jump is executed only for the final bit when it is a 1; for all other bit-1 iterations, the preceding jmp(x_dec) jumps back to BITLOOP and this instruction is never reached.

Parallel Operation

Enabling all four state machines with a single register write is what synchronizes them to the same clock cycle. After that, because the PIO program has a data-independent cycle count, they remain in lockstep indefinitely.

python 
PIO0_BASE = 0x50200000
CTRL      = PIO0_BASE + 0x00   # PIO0 control register

# Reset each SM's program counter to instruction 0
for sm in state_machines:
    sm.restart()

# Enable all four SMs in a single write — they execute their first cycle together
mem32[CTRL] = 0b1111

Direct FIFO writes

TX FIFO registers are physically adjacent 32-bit addresses in the RP2040's memory map. Writing to them with mem32 bypasses all MicroPython overhead and completes in a handful of CPU cycles (~8–12 ns each at 120 MHz), which is negligible compared to the 26.7 µs DShot600 frame period.

RegisterAddressState MachineGPIO
TXF00x50200010SM 0GPIO 6
TXF10x50200014SM 1GPIO 5
TXF20x50200018SM 2GPIO 4
TXF30x5020001cSM 3GPIO 3
⚠

Silent dropped writes. If a FIFO is full when you write to it, the write is silently discarded. The TX FIFO is 4 entries deep. At the packet rate used here (1 kHz timer, ~26.7 µs frame time), the FIFO is always drained well before the next write — but at high update rates, a dropped write to one FIFO causes that motor to repeat its previous packet, which looks like momentary cross-motor desynchronization.

Background Timer

The 1 kHz periodic timer is what decouples the DShot stream from Python execution. ESCs need a continuous stream of packets — any gap longer than ~20–100 ms (depending on firmware) causes the ESC to enter failsafe and cut motor output.

Before the timer was introduced, the DShot stream was driven by a while True loop calling send_dshot() directly. A single print() call inside that loop can block for hundreds of milliseconds when the USB-CDC output buffer fills — long enough to disarm every ESC.

python 
dshot_throttle = 0   # module-level global — readable by the timer ISR

def _dshot_ticker(timer):          # called 1000x/sec by the hardware timer
    packet = make_packet(dshot_throttle) << 16
    for addr in SM_TXFIFO:
        mem32[addr] = packet

dshot_timer = machine.Timer()
dshot_timer.init(
    freq=1000,
    mode=machine.Timer.PERIODIC,   # fires every 1 ms, indefinitely
    callback=_dshot_ticker,
)

# Main loop only needs to update the value — timer handles transmission
def main():
    global dshot_throttle
    for t in range(48, 600):
        dshot_throttle = t          # timer picks this up on its next tick
        print(f"throttle: {t}")     # safe to print — timer keeps stream alive
        time.sleep_ms(50)

ONE_SHOT vs PERIODIC

ModeBehaviorUse case
machine.Timer.PERIODIC Fires callback every period ms indefinitely until .deinit(). This driver. Keeps DShot stream continuous for the lifetime of the program.
machine.Timer.ONE_SHOT Fires callback exactly once after period ms, then stops. Delayed one-time actions, watchdogs, timeouts. Not suitable for continuous streaming.
✕

Callback signature must be exactly one parameter. machine.Timer calls its callback as callback(timer) — one argument, the timer object. A callback defined as def _dshot_ticker(timer, throttle) raises TypeError: function takes 1 positional argument but 2 were given inside the IRQ handler, 1,000 times per second. On RP2040, this floods the USB-CDC output buffer, effectively hanging the board.

The throttle value is read from the module-level dshot_throttle global — it does not need to be passed as a parameter.

Common Pitfalls

Throttle ramp starting from 1

⚠

DShot values 1–47 are reserved ESC commands (motor beep, direction change, etc.). Starting a ramp at 1 sends dozens of unintended commands before the motor ever receives a throttle value. Always start at 48.

print() inside a tight DShot loop

✕

When the board is not connected to an active REPL terminal, print() over USB-CDC can block for hundreds of milliseconds once the output buffer fills. A single blocked print() inside a bare while True: send_dshot() loop pauses the entire stream — long enough to disarm ESCs and cause exactly the "inconsistent motor movement" pattern. Move all logging outside the timer ISR and use the timer-based architecture so prints cannot affect signal continuity.

Two senders writing to the same FIFOs

✕

Do not call send_dshot() from both the main loop and a timer callback at the same time. The two callers write packets independently and interleave on the wire — the ESC alternately sees the loop's value and the timer's value, which looks like randomly fluctuating throttle. The timer should be the only source of FIFO writes. The main loop only updates dshot_throttle.

Timer callback with wrong signature

✕

machine.Timer calls its callback with exactly one argument. def _dshot_ticker(timer, extra_arg) crashes every tick with TypeError. The correct signature is def _dshot_ticker(timer) — read shared state from module-level globals.

Restarting a single state machine after synchronized start

⚠

Calling sm.restart() or sm.active(0) on one SM resets only that SM's program counter and clock divider phase. It will re-enter the loop out of sync with the other three. If an individual SM needs to be reset, restart all four and re-issue mem32[CTRL] = 0b1111.

Setting machine.freq() after creating state machines

⚠

The PIO clock divider is computed from the system clock at the moment rp2.StateMachine(..., freq=...) is called. Changing machine.freq() afterward does not update the divider, so the PIO will run at the wrong frequency. Always call machine.freq() before creating state machines.

✓

Scope check before debugging in software. If motors behave inconsistently after applying all of the above, probe GPIO 6 and GPIO 3 simultaneously on two oscilloscope channels. If the two waveforms are phase-locked and continuous throughout the run, the signal is correct — the issue is downstream (ESC calibration, power rail sag, mechanical imbalance, or per-ESC firmware settings). If the waveforms drift apart or show idle gaps, there is still a synchronization or stream-continuity issue to resolve in software.
